**Required sections (exact content to include):**

1. **Title:** "# MILP Formulation" (top-level heading)

2. **Overview paragraph:** The TETRA constraint optimization pipeline uses two MILP (Mixed-Integer Linear Programming) models in sequence. First, `ComputeAllocator` assigns computation nodes to hardware cores and time slots. Then, `TransferAndTensorAllocator` decides where tensors are stored and how data transfers are routed across the communication network. Both models are built through the solver facade (`SolverModel` ABC) and solved using either Gurobi or OR-Tools backends. The build pipeline for both follows the same pattern: prepare data, create variables, add constraints, set objective, solve. Source files: `stream/opt/allocation/constraint_optimization/allocation.py` (ComputeAllocator, 646 lines) and `stream/opt/allocation/constraint_optimization/transfer_and_tensor_allocation.py` (TransferAndTensorAllocator, 2155 lines).

3. **Two-Stage Allocation section (per D-04):** Explain how the two MILPs relate:
   - ComputeAllocator runs first: determines which computation node runs on which core, how many cores each node uses (k-split), and which time slot each node occupies. Its output (the allocation mapping) becomes a fixed input to the second stage.
   - TransferAndTensorAllocator runs second: given the fixed node-to-core placement, determines where each tensor is stored (on which memory cores), which transfer path routes data between cores, how data reuse levels are set, and how DMA channels are utilized. This is the larger and more complex MILP.
   - Both minimize total execution latency accounting for overlap (pipelining across iterations).

4. **ComputeAllocator section (per D-04):** Cover the CA MILP as a complete section. Include:

   **Build pipeline:** `__init__` calls `_prepare_constants()` which precomputes latencies, energies, dependency maps, node groups, weights, and capacities into a frozen `ComputeAllocatorConstants` dataclass. Then `_build_constraint_model(const)` runs: `_create_basic_sets` -> `_create_variables` -> constraint methods -> `_set_objective`.

   **Variable table for ComputeAllocator:** List the key variable families:

   | Variable | Type | Purpose |
   |----------|------|---------|
   | `k_vec[n,k]` | Binary | One-hot encoding of how many cores node `n` uses (k-split factor) |
   | `k_splits[n]` | Integer | Derived k-split count for node `n` (sum of k * k_vec) |
   | `core_asgn[c,n]` | Binary | Whether core `c` is assigned to node `n` |
   | `slot_asgn[s,n]` | Binary | Whether node `n` is assigned to time slot `s` |
   | `asgn[c,s,n]` | Binary | Whether node `n` runs on core `c` in slot `s` (3D assignment) |
   | `slot_idx[n]` | Integer | Derived slot index for node `n` |
   | `lat_id_core[n,c]` | Integer | Latency of node `n` on core `c` (selected from cost LUT) |
   | `lat_core_slot[c,s]` | Integer | Latency contribution of core `c` in slot `s` |
   | `lat_slot[s]` | Integer | Maximum latency across all cores in slot `s` |
   | `w_split[n]` / `w_core[c]` | Integer | Weight (capacity usage) per node and per core |
   | `idle_start[c,s]` / `idle_end[c,s]` | Binary | Idle period indicators for overlap computation |
   | `idle_sum[c]` | Integer | Total idle slot-latency for core `c` |
   | `lat_iter` / `idle_min` / `total_lat` | Integer | Per-iteration latency, minimum idle overlap, total latency |

   **Constraint summary for ComputeAllocator:** List each constraint method with a one-line purpose:
   - `_add_k_split_constraints`: Each node selects exactly one k-split factor (one-hot); k_splits is the derived integer value.
   - `_add_core_assignment_constraints`: Each core is assigned to at most one node per slot; number of cores assigned to a node equals its k-split factor.
   - `_add_slot_assignment_constraints`: Each node is assigned to exactly one slot; slot_idx is derived from slot_asgn.
   - `_add_group_constraints`: Nodes in the same "group" (e.g., same layer) must share the same k-split factor.
   - `_add_dependency_constraints`: If node A depends on node B, A's slot index must be strictly greater than B's.
   - `_add_weight_constraints`: Per-core capacity limits -- the combined weight of all nodes assigned to a core must not exceed the core's capacity.
   - `_add_latency_constraints`: Selects the correct latency from the cost LUT based on core assignment; slot latency is the max across all cores in that slot.
   - `_add_overlap_constraints`: Computes idle start/end indicators for each core-slot pair to determine pipelining overlap.

   **Objective:** `minimize total_lat = iterations * lat_iter - (iterations - 1) * idle_min`. The `lat_iter` is the sum of all slot latencies. The `idle_min` is the minimum total idle time across all cores, representing the overlap achievable through pipelining when the steady-state schedule repeats.

5. **TransferAndTensorAllocator Build Pipeline section (per D-01):** Explain the TTA build pipeline:
   - `__init__` categorizes data (tensors, transfers, slots, cores, reuse info) and calls `_build_model()`
   - `_build_model()` orchestrates: `_create_vars()` -> `_index_choice_metadata()` -> `_create_constraints()` -> `_overlap_and_objective()`
   - Each step is broken into private methods for modularity. Variable creation uses double-underscore helpers (`__create_tensor_placement_vars`, etc.). Constraint groups are each a separate single-underscore method.

6. **TTA Variable Families table (per D-02):** Include a variable family table:

   | Prefix | Variable | Type | Purpose |
   |--------|----------|------|---------|
   | `x_` | `x_tensor_choice[(t, choice)]` | Binary | Whether tensor `t` uses placement choice `choice` (a tuple of cores) |
   | `y_` | `y_path_choice[(tr, choice)]` | Binary | Whether transfer `tr` uses routing path `choice` (a MulticastPathPlan) |
   | `z_` | `z_stop[(t, stop)]` | Binary | Whether tensor `t`'s reuse stop level is `stop` (one-hot over reuse levels) |
   | `L_` | `slot_latency[s]` | Integer | Latency of time slot `s` |
   | `fires_` | `fires[tr]` | Integer | Number of times transfer `tr` fires per iteration (derived from reuse stop) |
   | `reuse_factor_` | `reuse_factors[tr]` | Integer | Reuse factor for transfer `tr` (derived from reuse stop) |

   Below the table, note that additional variables are created during overlap computation: `idleS`/`idleE` (binary idle indicators per resource per slot), `idle_lat` (integer idle latency per resource), `overlap` (integer, minimum idle latency across all resources), `link_used` (binary, whether a communication link is active), and DMA-related variables (`coreDmaIn`, `coreDmaOut`, `maxCoreDmaIn`, `maxCoreDmaOut`).

7. **TTA Constraint Groups section:** Document each constraint group called by `_create_constraints()` with a one-line to short-paragraph purpose. Organize into two sub-groups:

   **Always-active constraints (not guarded by ConstraintSelection):**
   - `_tensor_placement_constraints`: Each tensor selects exactly one placement choice (one-hot over x_ variables).
   - `_path_choice_constraints`: Each transfer selects exactly one routing path (one-hot over y_ variables), plus source/destination tensor coherence (path endpoints match tensor placements) and empty-path coherence.
   - `_transfer_fire_rate_constraints`: Defines `fires_` variables as a function of the reuse stop level (z_stop). The fire count determines how many times a transfer executes.
   - `_reuse_factor_rate_constraints`: Defines `reuse_factor_` variables as a function of the reuse stop level. The reuse factor captures the data reuse multiplier.
   - `_link_contention_constraints`: Each communication link can carry at most one active transfer per time slot (no link sharing within a slot).
   - `_slot_latency_constraints`: Slot latency is at least the latency of every transfer and compute operation assigned to that slot.
   - `_force_nonconstant_reuse_levels`: Non-constant tensors must have their reuse stop level set to a valid reuse point.
   - `_force_final_output_reuse_levels`: Final output tensors must use their full reuse range.
   - `_ensure_memory_and_compute_reuse_compatibility`: Memory-core reuse levels must be compatible with compute-core reuse patterns.
   - `_force_reuse_includes_spatial`: Reuse levels must include at least the spatial dimension.

   **ConstraintSelection-guarded constraints (per D-03):**
   Cross-reference `.claude/skills/optimization/constraint-selection.md` for the toggle-to-hardware mapping. Then document which methods are guarded:

   | ConstraintSelection Field | Guarded Method | Location | Effect When Disabled |
   |---------------------------|---------------|----------|---------------------|
   | `memory_capacity` | `_memory_capacity_constraints()` | `_create_constraints()` | Per-core memory usage no longer bounded |
   | `object_fifo_depth` | `_object_fifo_depth_constraints()` | `_create_constraints()` | Object-FIFO depth limits not enforced |
   | `buffer_descriptors` | `_buffer_descriptor_constraints()` | `_create_constraints()` | Buffer descriptor count limits not enforced |
   | `dma_channels` | `_add_dma_usage_constraints()` | `_overlap_and_objective()` | DMA channel limits not enforced AND DMA usage terms removed from objective |

   Note that DMA constraints are in `_overlap_and_objective()` rather than `_create_constraints()` because DMA accounting depends on the overlap/idle indicator variables. Also note that when `dma_channels` is disabled, the objective function simplifies to just `total_lat` (without DMA penalty terms). Each disabled group produces a WARNING log entry.

8. **TTA Overlap and Objective section:** Explain the overlap computation and objective:
   - `_overlap_and_objective()` computes pipelining overlap by tracking idle periods on each resource (links and cores)
   - For each resource, idle start/end indicators identify slots where the resource is unused before first use (prefix idle) and after last use (suffix idle)
   - Idle latency per resource is the sum of slot-latencies during idle periods, computed using binary-times-continuous linearization
   - The overlap variable is constrained to be at most each resource's idle latency (i.e., overlap = min of all idle latencies)
   - Total latency: `total_lat = iterations * sum(slot_latency) - (iterations - 1) * overlap`
   - Objective: `minimize total_lat + maxCoreDmaIn + maxCoreDmaOut` when DMA is enabled, or just `minimize total_lat` when DMA is disabled
   - The DMA terms in the objective act as a soft penalty, encouraging the solver to spread DMA usage evenly even though the hard DMA limits come from `NamespaceConstraints`

9. **Linearization Helpers section:** Document the linearization helper methods used throughout the MILP. Use Claude's discretion on detail level:
   - `_add_binary_product(a, b, base_name)`: Linearizes the product of two binary variables using three constraints (w <= a, w <= b, w >= a + b - 1). Used heavily in memory capacity and buffer descriptor constraints where two binary conditions must both hold.
   - `_add_binary_scaled_continuous(binary_var, continuous_var, continuous_ub, base_name)`: Linearizes z = binary * continuous using McCormick envelope (four constraints). Used in idle latency computation where slot latency (continuous) is scaled by idle indicators (binary).
   - `_add_binary_times_const_over_linexpr(...)`: Linearizes binary * (constant / linear_expression). For Gurobi backends, uses `add_genconstr_nl` for the division. For OR-Tools, uses piecewise linearization over discrete denominator values via `_add_const_over_discrete_denominators`. This is the only backend-dependent linearization path.

2. **Overview paragraph:** The namespace constraints system provides hardware-specific MILP constraint dispatch using the Strategy pattern. A `NamespaceConstraints` base class defines three overridable constraint methods (object-FIFO depth, buffer descriptors, DMA channels). Each hardware namespace (e.g., AMD AIE2) provides a subclass that overrides the methods relevant to its resource limits. The `TransferAndTensorContext` holds a tuple of namespace strategy instances and dispatches constraint calls to all of them. This design separates hardware-specific resource limits from the generic MILP formulation in `TransferAndTensorAllocator`. Source: `stream/opt/allocation/constraint_optimization/context.py`.

3. **NamespaceConstraints Base Class section:** Document the base class:
   - `NAMESPACE` class attribute: a string identifying the hardware namespace (e.g., `"aie2"`). Used by `applies_to()` to filter cores.
   - `applies_to(core)`: returns True if `core.namespace == self.NAMESPACE`. Constraint methods use this to iterate only over cores belonging to this namespace.
   - Three overridable methods (all no-ops in the base class):
     - `add_object_fifo_constraints(model, object_fifo_depth)`: enforces per-core object-FIFO depth limits. Parameters: the solver model and a dictionary mapping cores to their accumulated FIFO depth expressions.
     - `add_buffer_descriptor_constraints(model, buffer_descriptor_depth)`: enforces per-core buffer descriptor count limits. Parameters: the solver model and a dictionary mapping cores to their accumulated buffer descriptor expressions.
     - `add_dma_usage_constraints(model, dma_usage_in, dma_usage_out)`: enforces per-core DMA channel limits in both directions. Parameters: the solver model and two dictionaries mapping cores to their accumulated incoming/outgoing DMA usage expressions. Returns a list of DMA-related variables (empty list in base class).
   - The base class methods are no-ops so that a namespace subclass only needs to override the constraints relevant to its hardware. Methods left unoverridden remain silent no-ops.

4. **AIE2Constraints section:** Document the concrete subclass for AMD AIE2:
   - `NAMESPACE = "aie2"` -- targets AIE2 tile array cores.
   - Constructor accepts three DMA channel limit parameters (`compute_dma_channels`, `memory_dma_channels`, `shim_dma_channels`) plus `offchip_core_id`. These come from `CoreConstraintProfile` in config.py, with defaults: compute=8, memory=6, shim=2.

   **Object-FIFO depth:** For each core where `applies_to(core)` is True, constrains the accumulated FIFO depth expression to be at most `core.max_object_fifo_depth`.

   **Buffer descriptors:** For each qualifying core, constrains the accumulated buffer descriptor expression to be at most `core.max_object_fifo_depth` (the same hardware register governs both object-FIFO depth and buffer descriptor count on AIE2).

   **DMA channels:** DMA limits vary by tile type. Include a table:

   | Tile Type | Default Max DMA Channels | Identification |
   |-----------|--------------------------|----------------|
   | Compute | 8 | Cores with `CoreRole.COMPUTE` |
   | Memory | 6 | Cores with `CoreRole.MEMORY` |
   | Shim | 2 | Core matching `offchip_core_id` |

   Constraints apply to both S2MM (incoming) and MM2S (outgoing) directions independently. For each direction, the core's DMA usage expression must not exceed the tile-type-specific limit. The `get_max_dma_channels(core)` method selects the correct limit based on core role and ID.

5. **TransferAndTensorContext section:** Document the context class that orchestrates dispatch:
   - Frozen dataclass with key field: `namespace_constraints: tuple[NamespaceConstraints, ...]`
   - Three dispatch methods that iterate over all namespace strategies:
     - `add_object_fifo_constraints(model, object_fifo_depth)`: calls each strategy's FIFO constraint method
     - `add_buffer_descriptor_constraints(model, buffer_descriptor_depth)`: calls each strategy's buffer descriptor method
     - `add_dma_usage_constraints(model, dma_usage_in, dma_usage_out)`: calls each strategy's DMA method
   - The `TransferAndTensorAllocator` calls these dispatch methods (not the namespace strategies directly). This means the allocator is agnostic to which hardware namespaces are present.

6. **build_transfer_context Factory section:** Document how the context is assembled:
   - `build_transfer_context()` scans the accelerator's core list for known namespace strings
   - Currently only `"aie2"` is recognized -- if any core has `namespace == "aie2"`, an `AIE2Constraints` instance is added
   - DMA channel limits come from `CoreConstraintProfile` in the config (defaults: compute=8, memory=6, shim=2)
   - Future namespaces follow the same pattern: create a `NamespaceConstraints` subclass, detect the namespace string in `build_transfer_context()`, and append the instance to the tuple
   - This extensibility means adding support for a new hardware target only requires: (1) a new subclass of `NamespaceConstraints`, (2) a namespace detection block in `build_transfer_context()`

7. **ConstraintContext section:** Briefly document `ConstraintContext` (frozen dataclass in context.py) which provides the core lists used by `ComputeAllocator`:
